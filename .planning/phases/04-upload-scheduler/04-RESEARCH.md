## Summary

Phase 4 delivers three things: a `pipeline/upload.py` module that wraps YouTube Data API v3 and Instagram Graph API, a `run-cycle` CLI command that orchestrates the end-to-end flow, and cron-based scheduling. The technical work splits cleanly into YouTube upload (well-documented, mature Python library), Instagram upload (requires publicly accessible video URL — the key operational constraint), and metadata generation (Claude Haiku call added to the upload flow).

The YouTube upload path uses `google-api-python-client` (already in requirements.txt) and `google-auth-oauthlib` (needs adding). The OAuth flow uses `InstalledAppFlow.run_local_server()` for one-time setup, then serializes credentials to `youtube_token.json` for all subsequent runs. The `google.auth.transport.requests.Request()` auto-refresh handles access token expiry transparently. **Critical caveat**: YouTube API projects created after July 28, 2020 have videos locked to private until the API project passes a compliance audit. The `setup-youtube` command must warn the user and document this in CLAUDE.md.

The Instagram upload path uses the two-step container+publish Graph API flow. Videos must be served from a **publicly accessible URL** — Meta's servers fetch the file directly. For VPS deployment this is straightforward (serve from a temporary HTTP endpoint or cloud storage). For local dev this requires ngrok or Cloudflare Tunnel. This is a deployment-time concern, not a code problem, but it must be documented clearly.

**Primary recommendation:** Implement YouTube upload with `google-api-python-client` + `google-auth-oauthlib`, Instagram upload with `requests` + long-lived token stored in `instagram_token.json`, cron scheduling with direct venv python paths, and document the YouTube API audit requirement prominently.

### Pattern 1: YouTube OAuth One-Time Setup (InstalledAppFlow)

**What:** Opens a local browser tab, user consents, credentials serialized to JSON.
**When to use:** `python main.py --channel X setup-youtube` — run once per channel.

```python
# Source: https://googleapis.dev/python/google-auth-oauthlib/latest/reference/google_auth_oauthlib.flow.html
from google_auth_oauthlib.flow import InstalledAppFlow
import json, pathlib

SCOPES = ["https://www.googleapis.com/auth/youtube.upload"]

def setup_youtube_oauth(channel_cfg, token_path: pathlib.Path) -> None:
    client_config = {
        "installed": {
            "client_id": channel_cfg.youtube_client_id,
            "client_secret": channel_cfg.youtube_client_secret,
            "redirect_uris": ["http://localhost"],
            "auth_uri": "https://accounts.google.com/o/oauth2/auth",
            "token_uri": "https://oauth2.googleapis.com/token",
        }
    }
    flow = InstalledAppFlow.from_client_config(client_config, scopes=SCOPES)
    creds = flow.run_local_server(port=0)
    token_path.write_text(creds.to_json())
```

### Pattern 2: YouTube Upload with Auto-Refresh

**What:** Load saved credentials, auto-refresh if expired, call videos().insert().
**When to use:** Every `run-cycle` that uploads to YouTube.

```python
# Source: https://developers.google.com/youtube/v3/guides/uploading_a_video
from google.oauth2.credentials import Credentials
from google.auth.transport.requests import Request
from googleapiclient.discovery import build
from googleapiclient.http import MediaFileUpload

def upload_to_youtube(
    video_path: str,
    title: str,
    description: str,
    tags: list[str],
    token_path: pathlib.Path,
    client_id: str,
    client_secret: str,
) -> str:
    """Upload video; return YouTube video ID."""
    creds = Credentials.from_authorized_user_file(
        str(token_path),
        scopes=["https://www.googleapis.com/auth/youtube.upload"],
    )
    if creds.expired and creds.refresh_token:
        creds.refresh(Request())
        token_path.write_text(creds.to_json())  # save refreshed token

    youtube = build("youtube", "v3", credentials=creds)
    body = {
        "snippet": {
            "title": title,
            "description": description,
            "tags": tags,
            "categoryId": "22",   # People & Blogs — auto-classified as Short by YouTube
        },
        "status": {"privacyStatus": "public"},
    }
    request = youtube.videos().insert(
        part="snippet,status",
        body=body,
        media_body=MediaFileUpload(video_path, chunksize=-1, resumable=True),
    )
    response = _resumable_upload(request)   # handles retries + backoff
    return response["id"]
```

### Pattern 3: Instagram Reels Two-Step Upload

**What:** POST container with public video_url, poll status until FINISHED, then publish.
**When to use:** Every `run-cycle` that uploads to Instagram.

```python
# Source: https://developers.facebook.com/docs/instagram-platform/content-publishing/
import requests, time

GRAPH_BASE = "https://graph.instagram.com/v22.0"

def upload_to_instagram(
    video_url: str,          # MUST be publicly accessible
    caption: str,
    ig_user_id: str,
    access_token: str,
) -> str:
    """Upload Reel; return IG media ID."""
    # Step 1: create container
    resp = requests.post(
        f"{GRAPH_BASE}/{ig_user_id}/media",
        params={
            "media_type": "REELS",
            "video_url": video_url,
            "caption": caption,
            "share_to_feed": "true",
            "access_token": access_token,
        },
        timeout=30,
    )
    resp.raise_for_status()
    container_id = resp.json()["id"]

    # Step 2: poll until FINISHED (up to 5 minutes, once per minute)
    for _ in range(5):
        time.sleep(60)
        status = requests.get(
            f"{GRAPH_BASE}/{container_id}",
            params={"fields": "status_code", "access_token": access_token},
            timeout=10,
        ).json().get("status_code")
        if status == "FINISHED":
            break
        if status == "ERROR":
            raise RuntimeError(f"Instagram container {container_id} failed")
    else:
        raise RuntimeError("Instagram container timed out (>5 min)")

    # Step 3: publish
    resp = requests.post(
        f"{GRAPH_BASE}/{ig_user_id}/media_publish",
        params={"creation_id": container_id, "access_token": access_token},
        timeout=30,
    )
    resp.raise_for_status()
    return resp.json()["id"]
```

**What:** For local dev, serve the output video over a temporary HTTP server; use ngrok/Cloudflare Tunnel to get a public URL.

### Anti-Patterns to Avoid
- **Storing OAuth credentials in channels.yaml or .env**: Token JSON files belong in `data/channels/{slug}/` (gitignored). Client ID/secret go in channels.yaml (not secret); tokens are secret and gitignored.
- **Calling `source activate` in crontab**: Use absolute path to venv Python instead.
- **Blocking pipeline on Instagram container poll**: Use `time.sleep(60)` with max 5 iterations; raise after timeout so the caller can log and continue.
- **Setting privacyStatus to "public" without knowing API audit status**: Document the YouTube API audit requirement; videos may upload as private until the GCP project is audited.
- **Treating Instagram `video_url` as a local file path**: It must be a publicly accessible HTTP/HTTPS URL that Meta's servers can fetch.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YouTube OAuth token refresh | Custom refresh token HTTP call | `google.auth.transport.requests.Request()` + `creds.refresh()` | google-auth handles expiry, clock skew, retry |
| YouTube resumable upload with retry | Custom chunked upload loop | `MediaFileUpload(chunksize=-1, resumable=True)` + `_resumable_upload()` helper from official sample | Edge cases in partial upload resumption are complex |
| Instagram API HTTP calls | Custom session class | `requests.post()` / `requests.get()` directly | No SDK needed; it's REST |
| Cron daemon / scheduler | APScheduler or custom loop | System crontab | cron is already on every Linux machine; no process to crash |
| Credential storage | Custom encryption | gitignored JSON files in `data/channels/{slug}/` | Simple and sufficient for a personal pipeline |

**Key insight:** Both Google and Meta use standard REST + OAuth flows with mature Python libraries. The only custom code needed is the glue between them and the existing pipeline.

## Common Pitfalls

### Pitfall 1: YouTube Videos Locked as Private After Upload
**What goes wrong:** All YouTube API projects created after July 28, 2020 have uploads locked to private until the GCP project passes a YouTube API compliance audit.
**Why it happens:** YouTube policy change to prevent spam uploads via unaudited API projects.
**How to avoid:** Complete the YouTube API compliance audit at https://support.google.com/youtube/contact/yt_api_form. Until audited, test by setting `privacyStatus: "private"` deliberately and verifying the video appears in YouTube Studio.
**Warning signs:** Video uploads succeed (200 OK, returns video ID) but the video appears as private in YouTube Studio regardless of `privacyStatus: "public"` in the request body.

### Pitfall 2: Instagram video_url Not Publicly Accessible
**What goes wrong:** Container creation returns a container ID, but status polling returns `ERROR` — Meta could not fetch the file.
**Why it happens:** The `video_url` is a localhost path, internal network URL, or serves the wrong Content-Type.
**How to avoid:** For local dev, use `ngrok http 8000` before running the upload, pass the ngrok HTTPS URL. For VPS, ensure port is open and the path resolves to the .mp4 file. Video must be H264/MP4, 9:16 aspect ratio, 5-90 seconds for Reels tab eligibility.
**Warning signs:** Container status is `ERROR` within 1-2 polling cycles.

### Pitfall 3: Long-Lived Instagram Token Expiry
**What goes wrong:** Instagram access token silently expires after 60 days of non-use; all upload attempts return 401.
**Why it happens:** Long-lived tokens expire after 60 days unless refreshed. The refresh must be triggered proactively.
**How to avoid:** Store `expires_in` alongside the token in `instagram_token.json`. In `run-cycle`, check if token expires within 7 days and auto-refresh it via `GET https://graph.instagram.com/refresh_access_token?grant_type=ig_refresh_token&access_token={token}`.
**Warning signs:** Upload fails with `{"error": {"code": 190, "type": "OAuthException"}}`.

### Pitfall 4: YouTube Quota Exhaustion
**What goes wrong:** Daily quota (10,000 units) exhausted; further uploads return 403 `quotaExceeded`.
**Why it happens:** `videos.insert` costs 100 quota units per upload. 10,000 units / 100 = 100 uploads/day across ALL channels in the GCP project. With 3 channels × 2 uploads/day = 6 uploads/day (600 units) — well within limits. Only a risk if running many test uploads during development.
**How to avoid:** Use a dedicated GCP project per environment (dev vs production). Each project has its own 10,000-unit quota.
**Warning signs:** 403 errors with `reason: "quotaExceeded"` in the response JSON.

### Pitfall 5: `creds.from_authorized_user_file()` Requires `client_id` and `client_secret` in Token JSON
**What goes wrong:** `google.auth.exceptions.RefreshError` at token refresh time despite having a valid refresh token.
**Why it happens:** The `Credentials.from_authorized_user_file()` method needs `client_id` and `client_secret` in the token JSON file to refresh — but `InstalledAppFlow` only stores them if the flow was constructed with `from_client_config()` not just from scopes.
**How to avoid:** Always use `InstalledAppFlow.from_client_config(client_config, scopes=SCOPES)` (not `from_client_secrets_file`), and verify the saved JSON contains `client_id` and `client_secret` fields after `creds.to_json()`.
**Warning signs:** `google.auth.exceptions.RefreshError: ('No access token in response', {'error': 'invalid_client'})`.

### Pitfall 6: Cron Working Directory
**What goes wrong:** `main.py` fails with `channels.yaml not found` or `data/pipeline.db` errors in cron.
**Why it happens:** Cron runs with `HOME=/root` and no `PWD` — relative paths in config.py resolve against `/` not the project directory.
**How to avoid:** Always prefix cron entries with `cd /absolute/path/to/auto-shorts &&`. config.py uses `Path(__file__).parent` (absolute) so it handles this, but the `cd` is still needed for any relative path assumptions.

### Pitfall 7: Instagram IG User ID vs Instagram Handle
**What goes wrong:** `/{ig-user-id}/media` returns 400 — the handle (@username) was used instead of the numeric user ID.
**Why it happens:** Instagram Graph API requires the numeric IG Business account ID (e.g., `"17841400008460056"`), not the @handle.
**How to avoid:** Store `instagram_user_id` as a separate field in channels.yaml. Obtain it once via `GET https://graph.instagram.com/me?fields=id,username&access_token={token}` during `setup-instagram`.

## Code Examples

### Resumable Upload Helper (YouTube)
```python
# Source: https://developers.google.com/youtube/v3/guides/uploading_a_video
import random, time
from googleapiclient.errors import HttpError

RETRIABLE_STATUS_CODES = {500, 502, 503, 504}
MAX_RETRIES = 10

def _resumable_upload(insert_request) -> dict:
    response = None
    retry = 0
    while response is None:
        try:
            _, response = insert_request.next_chunk()
        except HttpError as e:
            if e.resp.status in RETRIABLE_STATUS_CODES:
                retry += 1
                if retry > MAX_RETRIES:
                    raise
                sleep = random.random() * (2 ** retry)
                time.sleep(sleep)
            else:
                raise
    return response
```

### Exponential Backoff Wrapper (existing project pattern)
```python
# Pattern: same as all other external calls in this project
import time, logging
logger = logging.getLogger(__name__)

def _with_retries(fn, label: str, max_retries: int = 3):
    for attempt in range(1, max_retries + 1):
        try:
            return fn()
        except Exception as e:
            if attempt == max_retries:
                raise
            wait = 2 ** attempt
            logger.warning("%s attempt %d/%d failed: %s — retrying in %ds",
                           label, attempt, max_retries, e, wait)
            time.sleep(wait)
```

| Old Approach | Current Approach | When Changed | Impact |

- `youtube.upload` scope-only: YouTube now recommends `youtube.force-ssl` for all write operations, but `youtube.upload` is still valid and narrower in scope (preferred for least-privilege).

## Open Questions

1. **Instagram public URL on local dev**
   - What we know: Meta requires a publicly accessible URL for `video_url`; local paths don't work.
   - What's unclear: The CONTEXT.md doesn't specify how the local file becomes a public URL. The pipeline currently saves videos to `output/{run_id}/final.mp4`.
   - Recommendation: The `run-cycle` command should document that Instagram upload requires a publicly reachable URL and either (a) skip Instagram silently if no public URL is configured, or (b) have `INSTAGRAM_PUBLIC_BASE_URL` in .env that the upload module prefixes. For VPS deployment, this is the VPS public IP + a simple route. This is a configuration matter, not a code blocker.

2. **YouTube API compliance audit**
   - What we know: New GCP projects (post-July 2020) upload videos as forced-private until audited.
   - What's unclear: Timeline for approval; whether personal/self-use projects are exempt.
   - Recommendation: Implement uploads with `privacyStatus: "public"` as intended, warn user in `setup-youtube` output, and note in CLAUDE.md. The pipeline is technically correct; the audit is an operational step.

3. **Instagram Business account requirement**
   - What we know: Graph API only supports Business or Creator Instagram accounts.
   - What's unclear: Whether the user's Instagram accounts are set up as Business accounts.
   - Recommendation: Document in `setup-instagram` output that the Instagram account must be a Business or Creator account connected to a Facebook Page.
